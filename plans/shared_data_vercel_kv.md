# План: Общие данные через Vercel KV

## Проблема

Сейчас данные хранятся в `localStorage` каждого браузера. Админ добавляет месяцы → они видны только админу. Гость видит только 11 месяцев из `initialData.js`. Нужно единое хранилище.

## Архитектура решения

```
┌─────────────────────────────────────────────────────┐
│                   Vercel KV (Redis)                  │
│              kv.get("kubarev_monthly_data")           │
│              kv.set("kubarev_monthly_data", {...})    │
└──────────────────────┬──────────────────────────────┘
                       │  REST API
┌──────────────────────▼──────────────────────────────┐
│        /api/data.js (Vercel Serverless Function)     │
│  GET  → возвращает JSON из KV                        │
│  POST → сохраняет JSON в KV (только админ)           │
│  PUT  → обновляет месяц в KV (только админ)          │
│  DELETE → удаляет месяц из KV (только админ)         │
└──────────────────────┬──────────────────────────────┘
                       │  HTTP fetch
┌──────────────────────▼──────────────────────────────┐
│            React Dashboard.jsx (фронтенд)             │
│  При загрузке: fetch GET /api/data → если KV пуст,   │
│                  fallback на initialData.js            │
│  Админ CRUD:    fetch POST/PUT/DELETE /api/data       │
│  localStorage:  больше НЕ используется для данных     │
└─────────────────────────────────────────────────────┘
```

## Этапы реализации

### Этап 1: Установка и настройка Vercel KV

- [ ] Установить `@vercel/kv` через npm
- [ ] Создать KV-базу в Vercel Dashboard (Settings → Storage → KV)
- [ ] Переменные окружения `KV_REST_API_URL` и `KV_REST_API_TOKEN` появятся автоматически в Vercel
- [ ] Создать файл `vercel.json` с настройками функций (если ещё нет)

### Этап 2: Создать API-эндпоинт /api/data.js

- [ ] Создать `api/data.js` — Vercel Serverless Function
- [ ] Реализовать методы:
  - `GET` — получает `kubarev_monthly_data` из KV (или возвращает `{}`)
  - `POST` — сохраняет полный объект данных (с проверкой пароля админа)
  - `PUT` — обновляет/добавляет один месяц (с проверкой пароля)
  - `DELETE` — удаляет один месяц (с проверкой пароля)
- [ ] Проверка пароля через заголовок `x-admin-token` (SHA-256 от пароля)

### Этап 3: Рефакторинг Dashboard.jsx — замена localStorage на KV API

- [ ] Добавить хук `useEffect` при старте: GET `/api/data` → заполнить `monthlyData`
- [ ] Если KV пуст (новый деплой), инициализировать его из `initialData.js` через POST
- [ ] `handleAddMonth` → PUT `/api/data` (с сохранением в локальный state)
- [ ] `handleDeleteMonth` → DELETE `/api/data` (с сохранением в локальный state)
- [ ] `handleOverwriteMonth` → PUT `/api/data`
- [ ] `handleImportBackup` → POST `/api/data` (полная замена)
- [ ] Бэкап/восстановление оставить как есть (локальный файл)

### Этап 4: Тестирование локально

- [ ] Настроить Vercel CLI локально: `vercel dev`
- [ ] Проверить: админ добавляет месяц → гость (инкогнито) видит изменения
- [ ] Проверить: `git push` → новый деплой → данные сохранены (KV переживает деплои)

### Этап 5: Деплой и финализация

- [ ] `git push` текущей версии на GitHub → авто-деплой на Vercel
- [ ] Обновить `DEPLOY.md` с инструкцией по настройке KV

---

## Безопасность

- Все мутации (POST/PUT/DELETE) требуют заголовок `x-admin-token` = SHA-256(`08250825`)
- Гости могут только GET (чтение)
- Ключ API не хранится на фронтенде (токен вычисляется при каждом запросе)
